About

문제를 직접 발견하고, 해결 방식도 스스로 설계하는 사람

CS에서 저는 항상 두 번 생각합니다. "이 케이스를 어떻게 처리할 것인가"와 "이게 왜 반복되는가". VOC가 쌓일 때 처리 속도보다 반복의 원인을 먼저 찾는 이유입니다. 173건을 전수 분류했더니 82%가 하나의 구조 문제에서 나왔고, 그 데이터로 개발팀을 설득해 당일 구현으로 CS 문의를 0건으로 만들었습니다.

쏘카의 CS는 예약·결제·차량 수령·반납·사고 대응까지 여정 전체에 걸쳐 있습니다. 어느 접점에서 VOC가 몰리는지, 그 원인이 앱 UX인지 정책 설계인지 운영 프로세스인지 — 구조를 보는 방식이 여기서도 그대로 작동한다고 생각합니다.

타 부서 인원들과 CS 팀을 운영한 경험, AI 도구를 직접 구현해 조직 내 공식 승인까지 받은 경험 — 둘 다 같은 원리에서 나왔습니다. 필요한 것이 없으면 기다리지 않고 직접 만듭니다.

판단 기준이 없으면 직접 만든다

CS 운영이 필요한 상황에서, 타 부서 인원들이 하나의 팀으로 움직일 수 있도록 판단 기준과 운영 구조를 먼저 설계했습니다. 기준을 만들면 팀은 그 기준을 따릅니다.

문제는 다시 생기지 않게 막는 것

CS 문의는 처리하는 게 아니라 구조로 차단합니다. VOC를 전수 분류해 원인을 규명하고, CX 개선과 운영 기준 수립으로 실행했습니다.

AI는 전략 수립의 도구

Chrome Extension 직접 구현으로 CS 8단계→2단계. AI를 효율화 도구가 아닌, 운영 구조를 설계하는 파트너로 씁니다.

CX 프로세스 개선 CS 입력 구조 설계 개발팀 협의

CS 입력 구조 재설계 — 고객 의도를 데이터로 만들어 재확인 왕복 95% 제거

처리 단계에서 고객에게 다시 묻는 것보다, 수집 단계에서 처음부터 제대로 받는 구조를 만드는 게 낫다고 판단했습니다. CRM 중복 필드를 재활용해 스펙을 직접 설계하고 개발팀과 협의·구현, AS 재확인 왕복을 95% 이상 없앤 사례입니다.

95%+ AS 재확인 왕복 감소 · CX 리드타임 단축
CS 운영 기획 기준 설계 운영 체계 수립

CS 운영 체계 수립 — 기준이 없던 곳에 직접 설계하다

CS 판단 기준, SLA 처리 기준, 에스컬레이션 가이드라인이 전무하던 환경에서 데이터와 계약 분석을 기반으로 운영 기준을 직접 수립했습니다. 3PL 손실 이슈를 구조화해 CEO/COO 직접 보고하고, 월 1,900만 정산 구조를 설계한 사례입니다.

월 1,900만 데이터 기반 정산 구조 설계 · CEO 승인
생성형 AI 활용 CS 자동화 운영 효율화

생성형 AI 활용 CS 자동화 — 8단계 프로세스를 2단계로

Claude로 Chrome Extension을 직접 구현해 CS 반복 프로세스를 8단계에서 2단계로 축소했습니다. 비개발 직군 유일하게 AI 도입 비용을 업무비로 공식 승인받았고, VOC 분류·계약 분석·보고 자동화를 실무에 적용한 사례입니다.

8→2단계 CS 자동화 · 1,151건 단독 처리
VOC 데이터 분석 CX 개선 실행 개발팀 협업

VOC 전수 분류 → CX 구조 문제 발굴 및 개선 실행

프로모션 오인 VOC 173건을 전수 분류해 82%의 동일 구조적 원인을 데이터로 규명했습니다. 원인 규명에서 그치지 않고 개발팀과 협의해 CX 개선으로 실행하고, 반복 CS 문의를 일 6~10건에서 0건으로 구조적으로 차단했습니다.
